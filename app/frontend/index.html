Verify button mapping

The index → physical button correspondence is auto-detected from your mouse's current button actions, not hardcoded — but it can be wrong if you've already remapped buttons away from their factory defaults. To confirm a marker for yourself:

  1. Pick a button whose identity you want to confirm.
  2. Set its action to Keyboard key → a key you don't normally use (e.g. KEY_F13).
  3. Run xev or showkey in a terminal, then physically press that button on the mouse.
  4. Whichever marker's key shows up is the physical button you pressed — relabel it below if it doesn't match.

Restore defaults

Resets every button — across all 5 onboard profiles — back to your confirmed working setup (left/right/middle click, standard Forward/Backward actions, Thumb Button's semicolon macro, wheel tilts, etc). This is a recovery net for profile drift or stray edits, not Logitech's out-of-box factory state. Doesn't touch DPI stages, polling rate, or your button labels/overrides.

Known quirks of this mouse
  • This mouse has 5 onboard profiles. Editing a button always edits whichever profile ratbagd currently reports as active — which has been observed to drift from what the hardware is actually running. If a change doesn't seem to take effect, use Restore defaults above to reset every profile to a known-good state.
  • Index 11 (factory action resolution-down) has no corresponding physical button on this mouse and is hidden from the UI.
  • RGB lighting and battery level aren't exposed here: the G502 X has no RGB, and ratbagctl's CLI has no battery-level command.
